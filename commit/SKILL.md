# Commit

Create standardized, semantic git commits using the [Conventional Commits](https://www.conventionalcommits.org/) specification. Analyze the actual diff to determine the appropriate type, scope, and message.

## Quick Reference

- Creates: Git commit with conventional message
- Requires: Staged or unstaged changes
- See: `skill.yaml` for full procedure detail

## Conventional Commit Format

```
<type>(<scope>): <subject>

<optional body — 1-2 sentences max>
```

## Commit Types

| Type | Purpose |
|---|---|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation only |
| `style` | Formatting/style (no logic) |
| `refactor` | Code refactor (no feature/fix) |
| `perf` | Performance improvement |
| `test` | Add/update tests |
| `build` | Build system/dependencies |
| `ci` | CI/config changes |
| `chore` | Maintenance/misc |
| `revert` | Revert a commit |

## Project Rules (MUST VERIFY)

- [ ] Title ≤ 50 characters
- [ ] Title is SPECIFIC (avoid: "improve", "enhance", "update")
- [ ] NO AI attribution (no "Co-Authored-By: Claude" or similar)
- [ ] NO ticket references in body (no "Fixes #123")
- [ ] NO emojis in commit message
- [ ] Body: 1-2 sentences max, or omit if title is clear
- [ ] Focus on WORK done, not file operations

## AI Attribution Policy

**STRICTLY FORBIDDEN** in commit messages:
- `Co-Authored-By: Claude`, `Co-Authored-By: GPT`, or any AI name
- 🤖 or any AI-related emojis
- "Generated by", "Created with", or similar phrases

Commits must appear as human-authored work.

## Workflow

### 0. Branch Validation (REQUIRED FIRST)

```bash
BRANCH=$(git branch --show-current)
# If branch is main or master — STOP and ask user to create a feature branch
```

Never commit directly to `main` or `master`.

### 1. Analyze Changes

```bash
git status --porcelain   # See all changes
git diff --staged        # If files are staged
git diff                 # If nothing staged
```

### 2. Stage Files (if needed)

Ask user what to stage; never auto-stage everything:

```bash
git add path/to/file1 path/to/file2
git add *.test.*
```

**Never stage**: `.env`, credentials, private keys, secrets.

### 3. Generate Commit Message

From the diff, determine:
- **Type**: What kind of change is this?
- **Scope**: What module is affected? (e.g., `contracts`, `tests`, `skills`, `config`)
- **Subject**: One-line summary (present tense, imperative mood)

### 4. User Review (ALWAYS confirm)

1. Show the generated commit message
2. Ask: "Commit with this message? (y/n)"
3. If **yes**: execute
4. If **no**: ask "Any changes I should make?" and revise

### 5. Execute Commit

```bash
# Single line
git commit -m "<type>(<scope>): <subject>"

# Multi-line (use HEREDOC for body)
git commit -m "$(cat <<'EOF'
<type>(<scope>): <subject>

<body — 1-2 sentences>
EOF
)"
```

## Examples

### Good commits
```
feat(contracts): add equivalence principle validation to escrow
fix(tests): handle mock_web timeout in price oracle test
docs(skills): add integration test environment comparison table
chore(deps): upgrade genlayer-test to fix GLSim connection issue
```

### Bad commits
```
chore(config): update configuration           ← vague
feat(api): improve API performance            ← vague
docs: add implementation plan to docs folder  ← file-focused, not work-focused
feat(app): add feature (Co-Authored-By: Claude) ← AI attribution
```

## Git Safety Protocol

- **NEVER commit to main/master** — create a feature branch first
- NEVER update git config
- NEVER run `--force` or hard reset without explicit user request
- NEVER skip hooks (`--no-verify`) unless user explicitly asks
- If a commit fails due to hooks: fix the issue and create a NEW commit (don't amend)
